# 贡献指南

欢迎参与 Hello Rust 教程的贡献！这份指南将帮助你了解如何改进文档、提交代码、以及成为项目的贡献者。

Hello Rust 是一个 Rust 学习资源，通过实际的代码样例帮助学习者掌握 Rust 的基础知识、高级特性以及框架应用。我们欢迎所有形式的贡献，无论是修复错误、改进文档、补充知识点，还是翻译工作。

---

## 如何改进文档

### 1. 发现错误

如果你在阅读文档时发现错误（拼写错误、代码错误、概念解释不准确等），请通过以下方式提交：

**提交 Issue**：
- 在 GitHub 上创建 Issue
- 标题格式：`[错误] 章节名称 - 错误类型`
- 描述错误内容、位置和复现方式
- 建议修复方案（如有）

**示例 Issue**：
```
标题：[错误] 所有权章节 - 代码示例无法编译

描述：
在 docs/src/basic/ownership.md 第 45 行的代码示例中，变量 s1 在所有权转移后仍然被使用，导致编译错误。

错误位置：
- 文件：docs/src/basic/ownership.md
- 行号：45-50

建议修复：
删除 println!("{}", s1); 这一行，或改为借用示例。
```

### 2. 改进示例

如果你发现代码示例可以改进（性能优化、更清晰的实现、更完整的注释），请提交 PR：

**提交 PR**：
- Fork 项目仓库
- 创建分支：`git checkout -b improve-example-章节名`
- 修改代码并确保可编译运行
- 添加或更新注释解释改进理由
- 提交 PR，标题格式：`[改进] 章节名称 - 改进内容`

**示例 PR 标题**：
```
[改进] 线程章节 - 使用 crossbeam 替代标准库通道
[改进] 所有权章节 - 添加内存布局 ASCII 图
```

### 3. 补充知识点

如果你有好的"小知识"（类比、真实案例、补充概念），欢迎提交：

**提交"小知识"PR**：
- 符合章节主题，不偏离核心内容
- 提供真实案例或贴近生活的类比
- 控制在 2-4 句话，简洁有趣
- 优先使用中国文化背景的类比

**示例"小知识"**：
```markdown
> 💡 **小知识**：所有权就像"图书馆借书"规则 - 一次只能借给一个人，其他人想看必须等归还或借复印件（克隆）。这样保证了不会有两个读者同时在书上乱涂乱画。
```

### 4. 翻译工作

我们计划在中文版完成后创建英文翻译版本。如果你有兴趣参与翻译工作：

**加入翻译组**：
- 在 GitHub Issue 中表达翻译意愿
- 指定负责的章节范围
- 遵循术语一致性（参考 glossary）
- 使用英文学术惯例，但保持风格一致

**翻译要求**：
- 术语表一致性（ownership → 所有权）
- 保持章节结构不变
- 代码注释保持英文（Rust 惯例）
- 链接同时提供英文和中文版本

## 代码要求

所有提交的代码必须符合以下标准：

### 必须满足

- **✅ 代码可编译运行**
  - 运行 `cargo build --workspace` 无错误
  - 代码示例在文档中标记版本信息
  
- **✅ 无 Clippy warning**
  - 运行 `cargo clippy --workspace` 无警告
  - 修复所有 Clippy 提示的改进建议
  
- **✅ 格式化代码**
  - 运行 `cargo fmt --all` 格式化代码
  - 遵循 Rust 标准格式化规范
  
- **✅ 包含测试（如适用）**
  - 为新功能添加单元测试
  - 测试覆盖核心逻辑
  - 测试代码在 `#[cfg(test)]` 模块中

### 代码示例来源

所有文档中的代码示例必须：

- **来自项目实际代码**（不要虚构示例）
- **提供 GitHub 链接**（指向真实文件）
- **标记源码位置**（文件路径和行号）
- **可独立编译运行**（读者可复制粘贴测试）

**正确示例**：
```markdown
## 代码示例

```rust
// 来自 src/basic/ownership_sample.rs
fn gives_ownership() -> String {
    let some = String::from("hello");
    some  // 返回值所有权转移给调用者
}
```

**完整源码**：[src/basic/ownership_sample.rs](https://github.com/savechina/hello-rust/blob/main/src/basic/ownership_sample.rs)
```

**错误示例**：
```markdown
❌ 不要使用虚构代码：
struct FakeExample {
    foo: String,  // 项目中没有这个结构体
}
```

### 验证命令

提交前请运行以下命令验证：

```bash
# 编译检查
cargo build --workspace

# Clippy 检查
cargo clippy --workspace

# 格式化
cargo fmt --all

# 测试运行
cargo test --workspace

# 文档构建
cd docs && mdbook build
```

## 文档要求

所有文档提交必须符合以下规范：

### 语言规范

- **✅ 使用简体中文**
  - 所有正文内容使用简体中文
  - 技术术语首次出现：中文（英文）
  - 后续使用：仅中文
  
- **✅ 解释清晰准确**
  - 使用平实语言，避免学术化表达
  - 不要使用"显然"、"简单"、"只需"等词汇
  - 每段一个概念，不超过 2-3 句话
  
- **✅ 符合文档风格**
  - 遵循章节模板结构
  - 使用鼓励性语气（"你可能会遇到这个错误 - 这很正常！"）
  - 使用第二人称"你"而不是"用户"

### 章节结构模板

每个章节必须包含以下部分：

1. **开篇故事** - 2-4 句话的类比或现实场景
2. **本章适合谁** - 1-2 句帮助读者自我评估
3. **你会学到什么** - 3-5 个具体学习目标
4. **前置要求** - 学习本章前需要的知识（链接到前置章节）
5. **第一个例子** - 最小的可运行代码（≤15 行）
6. **原理解析** - 使用简单语言和类比解释
7. **常见错误** - 2-3 个学习者常犯的错误
8. **动手练习** - 2-3 个小挑战，难度递增
9. **故障排查** - 基于症状的常见问题解决方案
10. **知识扩展** - 选学的高级提示（明确标记为可选）
11. **小结** - 3-5 个要点总结
12. **源码位置** - GitHub 链接指向实际文件

### 视觉规范

- **✅ 关键概念有 ASCII 图表**
  - 所有权、借用、生命周期必须有可视化图
  - 使用 `text` 语言标记代码块
  - 包含中文注释
  
- **✅ 标题层级清晰**
  - 使用标准 Markdown 标题层级
  - 每 200-300 字使用描述性标题
  - 不跳过标题层级（H1 → H2 → H3）
  
- **✅ 代码格式正确**
  - 使用 `rust` 语言标记
  - 包含注释解释关键行
  - 格式化符合 rustfmt 标准

## 审核流程

所有提交遵循以下审核流程：

### PR 审核流程

1. **提交 PR**
   - Fork 项目并创建分支
   - 提交修改并创建 Pull Request
   - PR 标题遵循格式规范
   
2. **CI 自动检查**
   - GitHub Actions 自动运行
   - 检查：编译、测试、Clippy、格式化
   - 检查：mdBook 构建、链接有效性
   - 检查失败会阻止合并
   
3. **维护者审核**
   - 维护者在 48 小时内审核
   - 检查内容质量、风格一致性
   - 提出改进建议或批准合并
   
4. **反馈修改或合并**
   - 根据反馈修改代码
   - 修改后重新触发 CI
   - 通过后合并到主分支

### 审核标准

维护者审核时会检查：

- **内容质量**：解释准确、代码正确、无误导
- **风格一致**：符合文档写作规范
- **完整性**：包含所有必需部分
- **可运行**：代码示例可编译执行
- **链接有效**：所有链接可访问（无 404）

## 如何报告问题

### 报告文档问题

如果你发现文档问题，请通过 GitHub Issues 报告：

1. **创建 Issue**
   - 标题格式：`[问题类型] 章节名称 - 简短描述`
   - 问题类型：错误、改进、建议、翻译
   
2. **提供详细信息**
   - 错误位置（文件路径、章节名）
   - 错误内容（截图或代码片段）
   - 复现方式（如何触发问题）
   - 建议方案（如有）

3. **问题标签**
   - `bug`：错误需要修复
   - `enhancement`：改进建议
   - `documentation`：文档相关问题
   - `good first issue`：适合新贡献者

### 报告代码问题

如果你发现代码问题：

1. **创建 Issue**
   - 标题格式：`[BUG] 文件路径 - 问题描述`
   - 提供：编译错误输出、运行结果
   
2. **提供环境信息**
   - Rust 版本（`rustc --version`）
   - 操作系统（macOS/Linux/Windows）
   - Crate 版本（如有依赖问题）
   
3. **提供复现步骤**
   - 最小化代码示例
   - 运行命令
   - 预期行为 vs 实际行为

## 翻译指南（未来规划）

### 翻译计划

中文版完成后，我们将启动英文翻译计划：

**Phase 1：术语表准备**
- 创建中英文术语对照表
- 确定技术术语标准翻译
- 维护一致性文档

**Phase 2：章节翻译**
- 按难度顺序翻译（基础 → 进阶 → 高级）
- 每位翻译者负责 1-2 个章节
- 保持结构一致性

**Phase 3：审校与发布**
- 技术审校（英文表达准确性）
- 教学审校（学习效果一致性）
- 双语版本同步发布

### 翻译要求

- **术语一致性**
  - 使用统一术语表
  - 首次出现提供原文
  - 避免多种翻译
  
- **结构一致性**
  - 保持章节结构不变
  - 保持标题层级不变
  - 保持示例位置不变
  
- **代码注释**
  - 代码注释保持英文
  - 符合 Rust 惯例
  - 不翻译变量名/函数名

### 加入翻译组

如果你有兴趣参与翻译工作：
- 在 GitHub Issue 中表达意愿
- 指定感兴趣的章节范围
- 提供翻译经验或语言能力证明
- 加入翻译讨论组

## 快速开始贡献

### 新贡献者指南

如果你是第一次贡献开源项目：

1. **阅读文档**
   - 阅读 README.md 了解项目
   - 阅读 docs/src/ 了解现有内容
   - 阅读本文 CONTRIBUTING.md
   
2. **选择任务**
   - 查看 GitHub Issues
   - 选择 `good first issue` 标签的问题
   - 从小改动开始（修复错误、改进注释）
   
3. **提交贡献**
   - Fork 项目
   - 创建分支
   - 修改内容
   - 提交 PR

### 常见贡献类型

| 贡献类型 | 难度 | 适合人群 | 示例 |
|---------|------|---------|------|
| 修复拼写错误 | 🟢 入门 | 所有贡献者 | 修正错别字 |
| 改进代码注释 | 🟢 入门 | Rust 学习者 | 添加中文注释 |

| 添加新章节 | 🔴 高级 | Rust 专家 | 编写高级主题 |

```bash

# 创建 Pull Request
