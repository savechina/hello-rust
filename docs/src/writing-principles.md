# 文档写作原则

**Purpose**: Guide for writing clear, accessible, and pedagogically effective Rust documentation in Chinese

**Created**: 2026-04-03  
**Branch**: `001-rust-tutorial-docs`  
**Applies to**: All chapters in `docs/src/`

## 本章适合谁

- **文档撰写者**: 将为本项目编写 Rust 教程章节的贡献者
- **技术编辑**: 负责审核文档质量的语言和技术准确性
- **社区贡献者**: 希望改进或扩展现有文档的 Rust 学习者

## 你会学到什么

完成本指南后，你可以：

1. 使用平实的中文撰写技术文档，避免知识诅咒陷阱
2. 为零基础学习者设计合适的解释方式
3. 应用认知负荷管理原则组织内容
4. 用类比和故事让抽象概念具体化
5. 创建鼓励性的学习环境，认可困难并庆祝进步

## 核心原则

### 原则 1：零基础友好 (Write for the "Almost Beginner")

**目标读者画像**:
- ✅ 有基础编程知识（变量、循环、函数）
- ❌ 零 Rust 经验
- ❌ 不熟悉系统编程概念

**写作检查清单**:

每次写下概念时，问自己：

1. **"如果我只懂 Python/JavaScript，能理解这句话吗？"**
   - ✅ Good: "Rust 的变量默认是不可变的，这意味着..."
   - ❌ Bad: "Rust 遵循线性类型系统，变量具有唯一所有权"

2. **"我是否假设了读者知道 Rust 特有概念？"**
   - ✅ Good: "借用 (borrowing) - 临时使用数据而不获取所有权"
   - ❌ Bad: "这个引用使用了可变借用"

3. **"我是否在段落开头就使用了术语堆砌？"**
   - ✅ Good: 先展示代码，再解释概念
   - ❌ Bad: 第一段就列出 5 个定义

**实践技巧**:
- 想象你在给一个聪明的朋友讲解，他刚学完 Python 基础
- 避免"显然"、"简单"、"只需"等词汇
- 第一次提到 Rust 特有概念时，必须提供解释或类比

---

### 原则 2：平实的语言 (Simple Language Over Technical Jargon)

**语言风格指南**:

| 避免 (Don't Use) | 使用 (Use Instead) |
|-----------------|-------------------|
| "显然，这个函数返回..." | "这个函数返回..." |
| "简单地说，所有权意味着..." | "所有权意味着..." |
| "只需调用 clone() 方法" | "我们可以调用 clone() 方法" |
| "如你所知，Rust 没有..." | "Rust 没有..." |
| "配置文件应该被更新" | "更新配置文件" |
| "可以被观察到" | "你可以看到" |

**句式要求**:
- 每句控制在**20 字以内**（不包括代码）
- 每段控制在**2-4 句话**
- 使用**主动语态**和**现在时**
- 使用第二人称"你"而不是"用户"或"开发者"

**示例对比**:

❌ **过于学术**:
> "所有权系统通过编译时静态分析确保内存安全性，防止数据竞争和悬垂指针等未定义行为。"

✅ **平实语言**:
> "Rust 用所有权系统管理内存。编译器在编译时检查你的代码，确保不会有数据竞争或悬垂指针。这意味着程序运行时更安全，不会崩溃。"

### 原则 3：小知识故事 (Knowledge Stories and Analogies)

**为什么需要故事**:
- 抽象概念难以记忆
- 类比连接已知和未知
- 故事让学习有趣

**优秀类比特征**:
- ✅ 来自日常生活（图书馆、借书、做饭）
- ✅ 中国文化背景（避免西方特定参考）
- ✅ 2-4 句话，简洁易记
- ✅ 准确反映技术本质

**经典示例**:

**所有权**:
> "就像借图书馆的书 - 一次只能借给一个人。如果别人借走了，你就不能同时借同一本书。"

**借用**:
> "就像把你的书借给朋友 - 你还是拥有者，但他们可以阅读。到期后书会还给你。"

**生命周期**:
> "就像图书馆的还书日期 - 告诉你借的书可以保留多久。超过期限就不能继续借阅。"

**创建你的类比**:
1. 识别核心概念（例如：独占访问）
2. 找日常生活中的相似场景（例如：图书馆、共享单车）
3. 测试类比是否准确（是否有误导性？）
4. 保持简洁（2-4 句话）

---

### 原则 4：渐进式披露 (Progressive Disclosure)

**信息组织原则**:

```
最简单示例 → 添加一个概念 → 再添加一个 → 完整示例
```

**章节结构示例**（以所有权为例）:

1. **第一步**: 展示最简单代码
   ```rust
   let x = 5;
   let y = x;
   println!("x = {}, y = {}", x, y); // 两者都能用
   ```

2. **第二步**: 引入 String 类型
   ```rust
   let s1 = String::from("hello");
   let s2 = s1;
   println!("s2 = {}", s2); // 可以
   // println!("s1 = {}", s1); // 编译错误！
   ```

3. **第三步**: 解释为什么（所有权转移）

4. **第四步**: 展示解决方案（克隆）

**禁止**:
- ❌ 在第一段就展示完整复杂示例
- ❌ 同时引入多个新概念
- ❌ 假设读者"应该知道"前置知识

---

### 原则 5：错误驱动学习 (Mistake-Driven Learning)

**为什么要展示错误**:
- 学习者一定会遇到编译器错误
- 提前展示减少挫败感
- 错误是最好的老师

**错误展示格式**:

```rust
let s1 = String::from("hello");
let s2 = s1;
println!("{}", s1); // ❌ 编译错误！
```

**编译器输出**:
```
error[E0382]: borrow of moved value: `s1`
  --> src/main.rs:4:20
   |
2  |     let s2 = s1;
   |              -- value moved here
3  |     println!("{}", s1); // ❌ 编译错误！
   |                    ^^ value borrowed here after move
```

**解释错误**:
> "编译器告诉我们 `s1` 在第 2 行已经移动给 `s2` 了，所以第 3 行不能再用 `s1`。这是 Rust 防止悬垂指针的方式。"

**错误章节必备**:
1. 真实编译器错误消息
2. 中文解释错误原因
3. 修复方法（至少一种）
4. 为什么 Rust 要这样设计

---

### 原则 6：具体先于抽象 (Concrete Before Abstract)

**教学顺序**:

```
✅ 正确顺序:
1. 可运行的代码
2. 运行结果
3. 原理解释
4. 变体和边界情况

❌ 错误顺序:
1. 理论定义
2. 抽象概念
3. 代码示例
4. 练习
```

**示例**（讲解函数）:

✅ **具体先于抽象**:
> "让我们看一个函数:
> 
> ```rust
> fn greet(name: &str) -> String {
>     format!("Hello, {}!", name)
> }
> 
> let message = greet("Alice");
> println!("{}", message); // 输出：Hello, Alice!
> ```
> 
> 这个函数接受一个名字，返回问候语。现在让我们看看它的组成部分..."

❌ **抽象先于具体**:
> "函数是带有参数和返回类型的可重用代码块。在 Rust 中，函数使用 fn 关键字定义..."

## 语气基调

### 鼓励性语气

**使用**:
- ✅ "你可能会遇到这个错误 - 这很正常！"
- ✅ "太棒了！你完成了第一个所有 权示例！"
- ✅ "让我们看看会发生什么..."
- ✅ "这个概念很有挑战性 - 很多学习者在这里遇到困难"

**避免**:
- ❌ "如你所知..."
- ❌ "显然..."
- ❌ "简单..."
- ❌ "只需..."
- ❌ "你应该..."

### 认可难度

**诚实面对挑战**:

> "Rust 的所有权系统与大多数语言不同。如果你第一次没理解，这完全正常。许多学习者在第 3 遍、第 4 遍阅读时才恍然大悟。坚持下去！"

> "借用检查器可能会让你觉得'编译器在跟我作对'。实际上它在帮你避免难以调试的错误。学会与它合作，而不是对抗。"

## 章节结构模板

**每章必须包含 12 个部分**:

1. **开篇故事** (2-4 句话的类比)
2. **本章适合谁** (1-2 句自我评估)
3. **你会学到什么** (3-5 个具体目标)
4. **前置要求** (链接到前置章节)
5. **第一个例子** (<15 行代码)
6. **原理解析** (简单语言解释)
7. **常见错误** (2-3 个编译器错误)
8. **动手练习** (2-3 个挑战)
9. **故障排查** (FAQ)
10. **知识扩展** (选学高级内容)
11. **小结** (3-5 个要点)
12. **术语表** (复杂章节需要)

## 质量检查清单

**完成章节后，逐一检查**:

### 语言质量
- [ ] 没有使用"显然"、"简单"、"只需"等词汇
- [ ] 每句控制在 20 字以内
- [ ] 每段控制在 2-4 句话
- [ ] 使用"你"而不是"用户"或"开发者"
- [ ] 技术术语首次出现有中文 (英文) 格式

### 教学法质量
- [ ] 开篇有小故事或类比
- [ ] 先展示代码，再解释原理
- [ ] 包含至少 2 个编译器错误示例
- [ ] 有错误原因解释和修复方法
- [ ] 包含 2-3 个动手练习

### 技术质量
- [ ] 所有代码示例可以编译
- [ ] 标明了完整示例的文件路径
- [ ] 有前置知识章节链接
- [ ] 提到相关外部资源（Rust Book 等）
- [ ]  unsafe 代码有明确警告

## 小结

**记住 6 个核心原则**:

1. **零基础友好** - 假设读者聪明但零 Rust 经验
2. **平实的语言** - 避免学术腔调，用日常语言
3. **小知识故事** - 用类比让抽象具体化
4. **渐进式披露** - 从简单到复杂，一次一个概念
5. **错误驱动学习** - 展示真实错误，解释原因
6. **具体先于抽象** - 代码先行，理论后解释

**语气始终**:
- 鼓励性，不要说教
- 认可难度，庆祝进步
- 使用"我们"而不是"你"

**最终目标**:
让学习者在遇到困难时想"这个文档懂我的挣扎"，而不是"写文档的人太傲慢了"。

---

**文档质量承诺**:
> "我们写给那个昨天还在为所有权困惑、今天想再试一次的学习者。如果我们的文档能让他少一点挫败、多一点信心，那就是成功。"
